# CognID-GenRec 待解决问题

> 最近更新：2026-07-11  
> 排序原则：先保证数据与评估可信，再保证训练/推理一致，之后优化模型效果和工程成熟度。

## P0：实验可信度与核心闭环

### P0-1 建立 Evaluation Protocol V2

**当前状态**

未实现。当前 small matrix leave-one-out、baseline candidate 和 ANN candidate 不在同一候选空间，实验面板只能作为诊断结果。

**证据**

- KuaiRec small matrix 密度 99.6%，每个用户几乎看过全部 3,327 个视频。
- 本地 sequence catalog 为 3,327，ANN catalog 为 10,731。
- 排除 train history 后，baseline 平均剩余 14.63 个候选，ANN 平均剩余 7,418.63 个候选。
- 随机 Top-10 的期望命中约为 0.7307，当前 Popular HR@10 为 0.7123。
- Popular/ItemCF 的 `fit()` 当前接收包含 validation/test 的完整 `item_ids`，即使只读对照表明这不是 ItemCF 高分主因，也仍不符合严格的 train-only 拟合边界。

**建议设计**

建立两条明确分离的实验协议：

1. `sequential_next_item`：使用 big matrix 按时间构建序列；所有模型只在 train 事件上拟合；validation 用于选模型；test history 可以包含 validation 已观察事件；所有方法使用同一个 candidate manifest。
2. `fully_observed_preference`：使用 small matrix 的 `watch_ratio` 作为显式或分级相关性标签，对统一的 3,327 item universe 做全排序/离线偏好评估，不再把 3,300 多条历史全部排除后只预测最后一个 item。

每个 run 必须记录：

- dataset version、matrix、日期范围和采样规则；
- train/validation/test 时间边界；
- candidate universe、target reachability 和历史过滤规则；
- full-sort 或 negative-sampling 模式；
- random seed 和配置版本。

**计划落点**

- `src/cognid_genrec/kuairec/sequences.py`
- `src/cognid_genrec/evaluation/segmented_metrics.py`
- `scripts/prepare_kuairec_data.py`
- `scripts/evaluate_kuairec_retriever.py`
- 建议新增 `src/cognid_genrec/evaluation/protocol.py`
- 建议新增 `scripts/audit_kuairec_protocol.py`

**验收标准**

- Popular、ItemCF、ANN 接收同一个持久化 candidate manifest。
- 自动检查每个 test target 都在 candidate universe 中。
- baseline 只能读取 train 数据拟合。
- 同时提供 `sequential_next_item` 与 `fully_observed_preference` 报告，报告标题明确协议名称。
- 增加回归测试：candidate 不一致、target 不可达、训练集包含 test target 时必须失败。
- 修正后的指标覆盖旧面板，并在文档中保留旧指标为“已废弃诊断结果”。

### P0-2 打通真实 Retriever 训练—评估—服务闭环

**当前状态**

未实现。训练会保存 `SequenceEncoder`，但离线评估和 FastAPI 服务没有加载它，当前 query 是历史 item embedding 的平均值。

**证据**

- 训练保存 `encoder_state_dict`、`item_embedding_state_dict` 和 config：`scripts/train_kuairec_retriever.py`。
- 离线评估 `ann_recommendations()` 只读取 `ann_items.npy` 并平均历史 item vector。
- `KuaiRecRecommendationEngine._query_embedding()` 同样只平均 item vector。
- API 在 artifact 缺失时静默返回内置 sample engine，容易让演示模式与真实模型模式混淆。

**建议设计**

- 抽出统一 `RetrieverArtifact` / `KuaiRecRetriever`，负责 checkpoint schema、encoder 恢复、item mapping、action mapping、ANN index 和 model metadata。
- 离线评估与 API 必须调用同一个 `encode_user(history_items, actions, time_deltas)`。
- artifact 缺失或版本不兼容时默认 fail fast；仅显式 `sample_mode=true` 时允许样例引擎。
- 修复长序列 item/action/time suffix 对齐：三者先按原事件对齐，再统一取最后 `max_len`。

**计划落点**

- `scripts/train_kuairec_retriever.py`
- `scripts/evaluate_kuairec_retriever.py`
- `src/cognid_genrec/models/sequence_encoder.py`
- `src/cognid_genrec/service/kuairec_api.py`
- 建议新增 `src/cognid_genrec/models/retriever.py`
- 建议新增 `src/cognid_genrec/artifacts/retriever_artifact.py`

**验收标准**

- evaluation 和 API 都能从 checkpoint 恢复 encoder 并生成相同 user embedding。
- action 或 time delta 改变时，测试能观察到 user embedding 或 Top-K 结果变化。
- 长度超过 `max_len` 的测试断言 item/action/time 来自相同事件后缀。
- API 返回真实 `model_version`、`checkpoint_id` 和 `candidate_version`。
- 禁止仍以均值 item embedding 的结果命名为 `ann_transformer`。

### P0-3 改进训练样本构造与对比学习稳定性

**当前状态**

未实现。当前每个用户仅构造一个 validation-target 训练样本，1,411 个用户只产生 23 个 batch；输出 `avg_loss=276.1961`，尚无 validation early stopping、训练曲线或可复现实验配置。

**外部参考**

- [GenRec](https://github.com/phonism/genrec) 对 train prefix 使用 sliding-window 样本，同时保留最后两条给 validation/test。
- [Meta Generative Recommenders](https://github.com/meta-recsys/generative-recommenders) 在序列多个位置执行自回归监督，配置 L2 normalization、sampled softmax、temperature 和候选索引。

**建议设计**

- 对每个用户的 train prefix 生成窗口样本，可配置 `stride`、`max_windows_per_user` 和最近窗口偏置，避免一次性展开全部 460 万事件造成内存压力。
- user/item embedding 在 InfoNCE 前做 L2 normalization。
- 识别 batch 内重复 target，避免把相同正例当成假负例。
- 支持 catalog sampled negatives 或 popularity-aware/hard negative sampling。
- 增加 seed、device、gradient clipping、scheduler、validation loop、early stopping 和 best checkpoint。
- 评估不同 action 权重目标，区分曝光/短播与高兴趣，不把所有曝光等价视为正反馈。

**计划落点**

- `scripts/train_kuairec_retriever.py`
- `src/cognid_genrec/models/contrastive.py`
- `src/cognid_genrec/models/sequence_encoder.py`
- 建议新增 `src/cognid_genrec/data/training_examples.py`
- 建议新增训练配置文件，例如 `configs/retriever/kuai_big.yaml`

**验收标准**

- 训练样本数、用户数、窗口规则、正负样本分布写入 run manifest。
- 固定 seed 重跑的样本划分与首轮指标一致。
- validation 指标驱动 best checkpoint，test 仅在模型确定后运行。
- loss、embedding norm、重复 target 比例和各 action 样本占比可观测。
- 至少包含 mean-item、Popular、ItemCF、当前 Transformer 的统一协议对照，以及 action/time/negative-sampling 消融。

### P0-4 实现 Semantic ID V2 与冲突安全评估

**当前状态**

未实现。当前 KMeans Semantic ID 唯一值只有 198，冲突率 98.15%，不能安全映射回具体 item。

**证据**

- `SemanticIDTokenizer` 的每层 KMeans 独立聚类同一份特征，不是 residual hierarchy。
- 当前 `reports/metrics/kuairec_semantic_id_kmeans.md` 已记录总体及 head/middle/tail 冲突率。

**外部参考**

- [GRID](https://github.com/snap-research/GRID)：Residual KMeans/RQ-VAE/RVQ、冲突追加消歧位、SID 生成模型。
- [GenRec](https://github.com/phonism/genrec)：sliding-window TIGER、Semantic ID collision logging 和 disambiguation suffix。
- [OpenOneRec](https://github.com/Kuaishou-OneRec/OpenOneRec)：Residual KMeans Itemic Token，以及 SID/PID 双口径推荐评估。

**建议设计**

- 将普通 KMeans 主线替换为 residual KMeans，或完善现有 RQ-VAE；每层量化上一层 residual。
- 根据 10k 级 catalog 校准 `num_levels` 和 `codebook_size`，不直接照搬面向大模型的 8192 codebook。
- 对共享 SID 的 item 追加稳定 disambiguation token，保留原层级 token 用于语义解释。
- 同时评估 reconstruction error、codebook utilization/perplexity、unique ratio、collision group size、topic purity，以及 head/middle/tail 分布。
- 生成式推荐必须同时报告 SID-level 和 PID/item-level Recall/NDCG，不能仅按共享 SID 判命中。

**计划落点**

- `src/cognid_genrec/tokenizer/semantic_id_tokenizer.py`
- `scripts/train_kuairec_semantic_id.py`
- `src/cognid_genrec/retrieval/semantic_id_decoder.py`
- `tests/test_semantic_id_tokenizer.py`

**验收标准**

- Semantic ID 可重复生成，item 到最终 token 序列一一映射。
- 最终 collision rate 为 0；同时保留追加消歧前的原始语义冲突指标。
- 报告 codebook 利用率和最大 collision group，而不仅是唯一 ID 数。
- 新增冲突数据回归测试，保证 SID-level 命中不会错误计作多个 PID/item 命中。

## P2：高级模型与平台扩展

### P2-1 HSTU / 更长序列建模

可以学习 Meta Generative Recommenders 的 HSTU、sampled softmax 和高效 attention，但只有在 Evaluation Protocol V2、真实 encoder 推理和稳定训练全部完成后才有比较价值。

**暂不作为 P0 的原因**：当前主要瓶颈是实验协议和链路不一致，不是 Transformer block 不够先进；直接迁移 HSTU 还会引入较重的 CUDA/Triton/TorchRec 依赖。

### P2-2 LLM/Itemic Token/多任务生成式推荐

可以参考 OpenOneRec 的 item understanding、label-conditioned recommendation、interactive recommendation 和 explanation 分层任务，但不应直接引入 Qwen3 1.7B/8B、RL 或 distillation 作为当前主线。

**前置条件**：

- Semantic ID V2 达到 PID/item 级安全映射；
- 有统一 benchmark 和可重复 baseline；
- 有可用 GPU、模型权重和训练资源；
- 能明确区分传统 next-item、生成 SID、文本解释三个任务。

### P2-3 大规模训练与部署环境

在 big matrix 采样、checkpoint/resume 和 run manifest 稳定后，再考虑多 GPU、Docker/Apptainer、Faiss 持久化索引、批量推理和更完整的监控。

## 暂时不要做

- 不要用当前 ItemCF 0.9865 与 ANN Transformer 0.0007 证明传统方法明显优于生成式模型。
- 不要把当前均值 item embedding 推理路径继续命名为真实 Transformer 效果。
- 不要在 Semantic ID 冲突率 98.15% 时只报告 SID-level 命中。
- 不要在协议修正前通过增加 epoch 或更换 HSTU/LLM 来追求面板数字。
- 不要把离线指标写成线上 CTR、观看时长、收入或生产效果。

## 推荐实施顺序

1. P0-1 Evaluation Protocol V2。
2. P0-2 Retriever 训练—推理一致性与长序列特征对齐。
3. P0-3 sliding-window、归一化和负采样。
4. P0-4 Semantic ID V2 与 SID/PID 双评估。
5. P1-1 时间安全特征。
6. P1-2 Run/Artifact 管理。
7. P1-3 API 与 ANN 工程化。
8. P1-4 CI 与端到端回归。
9. 资源允许时再进入 P2。
